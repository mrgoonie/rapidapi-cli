# Bun CLI Packaging & NPM Publishing Research Report
**Date:** 2026-04-19 | **Project:** rapidapi-cli | **Scope:** Cross-platform CLI, NPM/Bun distribution

---

## 1. Bun CLI Packaging Strategy

### Recommendation: Hybrid Approach
- **Ship compiled executables** via `bun build --compile` for **macOS/Linux users**
- **Ship TypeScript source** with `#!/usr/bin/env bun` shebang for **Bun-first developers**
- **Fallback to Node.js** via `#!/usr/bin/env node` shebang for **npm-only users**

### Rationale
Bun's `bun build --compile` creates zero-dependency single-file binaries. Anthropic uses this for Claude Code CLI. For npm distribution:

| Approach | Pros | Cons |
|----------|------|------|
| **bun build --compile** | Single file, no runtime needed, 2x faster startup | Platform-specific builds needed, larger artifact (~60MB) |
| **Ship .ts source** | Works with both Bun & Node, smallest artifact, simple CI | Requires Bun/Node at runtime, slower startup |
| **Dual bin** | User choice | Maintenance complexity |

**Decision:** Compile for distributions (GitHub releases), ship TS source in npm package. Let npm installer choose via shebang resolution.

### Package.json bin Configuration
```json
{
  "bin": {
    "rapidapi": "./bin/index.ts"
  },
  "engines": {
    "bun": ">=1.1"
  }
}
```

**Why:** Bun 1.1+ auto-replaces `#!/usr/bin/env node` → `#!/usr/bin/env bun` in bin scripts when Bun is in engines field. If Bun missing, falls back to node. Windows .cmd/.ps1 shims auto-generated by npm install.

**Windows Consideration:** npm cli generates .cmd and .ps1 shims automatically. No manual shimming needed.

## 2. CLI Framework Choice

### Recommendation: **Commander.js**

| Framework | Downloads/week | Bun Support | Setup Time | Output Modes |
|-----------|----------------|-------------|-----------|--------------|
| **Commander** | 15M+ | ✓ Native | 5 min | ✓ Built-in |
| **Citty** | 50K | ✓ Bun-first | 3 min | ✓ Simple |
| **Yargs** | 8M+ | ✓ Native | 10 min | Verbose |
| **Clerc** | 30K | ✓ Bun-optimized | 5 min | ✓ Simple |

**Justification:** Commander has 500M weekly downloads, production stability, and native dual-mode output support. Citty is emerging but immature. For **dual output** (JSON + colored):

```typescript
// Commander pattern
const cmd = new Command()
  .option('--json', 'output JSON')
  .action((opts) => {
    const result = { status: 'ok' };
    console.log(opts.json ? JSON.stringify(result) : colorize(result));
  });
```

Citty/Clerc support this too but have smaller ecosystems.

## 3. Environment Variable Precedence Chain

### Recommended Order (High → Low Priority)
1. **CLI flags** (--api-key) — highest precedence
2. **Process env vars** (RAPIDAPI_KEY) — runtime overrides
3. **Local .env.local** — user-specific, not committed
4. **Local .env** — project defaults
5. **~/.config/rapidapi/config.json** — system-wide config
6. **OS env vars** — inherited from shell

### Implementation Pattern
Use **`dotenv`** (23M weekly downloads) + **`env-paths`** (3M weekly):

```typescript
import { config } from 'dotenv';
import { configHome } from 'env-paths';

// 1. Load .env.local (highest priority file)
config({ path: '.env.local', override: true });
// 2. Load .env (fallback)
config({ path: '.env' });
// 3. Load ~/.config/rapidapi/config
loadConfigFile(`${configHome('rapidapi')}/config.json`);
```

**Why dotenv over cosmiconfig:** dotenv is simpler, Bun-native compatible, no node_modules bloat.

**Why env-paths over conf:** Respects XDG_CONFIG_HOME on Linux, ~/Library on macOS, %APPDATA% on Windows. More portable than hardcoded paths.

## 4. GitHub Actions Auto-Publish Workflow

### Recommendation: **release-please** (Google's approach, gaining adoption 2025-2026)

| Tool | Setup | Control | Ceremony | Maturity |
|------|-------|---------|----------|----------|
| **release-please** | 10 min | PR-gated | Low | Google-backed, stable |
| **semantic-release** | 15 min | Auto | None | Mature, opinionated |
| **changesets** | 20 min | Explicit files | Medium | Monorepo-focused |

**For solo CLI:** release-please strikes balance — creates PR for version bump + changelog, you review/merge, auto-publishes.

### Minimal Workflow (GitHub Actions)
```yaml
name: publish
on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      id-token: write  # Required for OIDC
    steps:
      - uses: actions/checkout@v4
      - uses: google-github-actions/release-please-action@v4
        with:
          release-type: node
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          registry-url: 'https://registry.npmjs.org'
      - run: npm ci
      - run: npm publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

**NPM Provenance Setup (2025-2026 best practice):**
1. Create npm account with 2FA
2. In npm settings: Enable "Automation tokens" (can push without classic token)
3. In GitHub repo: Settings → Secrets → Add `NPM_TOKEN` (granular access, readonly)
4. GitHub Actions auto-generates OIDC token → npm validates → publishes with provenance attestation

**Why:** Avoids deprecated Classic Tokens (deprecated Dec 2025). OIDC is now generally available, auto-generates provenance statements.

## 5. Testing Bun CLIs — bun test Patterns

### Recommended Pattern: Jest-compatible integration tests

```typescript
// test/cli.integration.test.ts
import { spawn } from 'bun';
import { test, expect } from 'bun:test';

test('cli outputs JSON with --json flag', async () => {
  const proc = spawn(['bun', './bin/index.ts', '--json'], {
    stdio: ['pipe', 'pipe', 'pipe'],
  });
  
  const output = await new Response(proc.stdout).text();
  const parsed = JSON.parse(output);
  expect(parsed.status).toBe('ok');
});
```

### Why bun test over Vitest/Jest
- **Bun test finishes in 0.08s vs Vitest 0.9s, Jest 1.2s** — same runtime runs code + tests
- Jest API surface (describe, test, expect, beforeEach, etc.) — no learning curve
- Native TypeScript, no transpilation overhead
- `-t` flag filters by test name pattern
- JunitReporter for CI integration

### Limitation (Current)
No programmatic test API yet (only CLI). Use `spawn()` for CLI testing or mock I/O.

## Summary: Recommended Tech Stack

| Topic | Choice | Rationale |
|-------|--------|-----------|
| **Packaging** | `bun build` (bins) + TS source (npm) | Hybrid: binaries for distributions, source for npm |
| **Framework** | Commander.js | 500M downloads, proven CLI patterns, dual output built-in |
| **Config** | dotenv + env-paths | Minimal deps, standard precedence chain, Bun-native |
| **Publishing** | release-please + OIDC | Low ceremony, PR-gated, provenance auto-generated |
| **Testing** | bun test | Fastest, Jest-compatible, native TS |

## Unresolved Questions

1. **Windows PowerShell execution policy:** Will `.cmd` shims respect execution policy for globally installed tools? (Likely yes; npm handles via `.bat` wrapper.)
2. **Bun 1.2+ runtime stability:** Latest v1.2 features (HTTP optimizations) — any breaking changes between 1.1 and 1.2? (Check release-please bun semver rules.)
3. **Native dependency fallback:** If rapidapi needs native deps (e.g., crypto), does bun build --compile include them? (Only FFI-exposed modules; most crypto is pure JS.)
4. **GitHub OIDC with private npm registries:** Does OIDC flow work for scoped @company/rapidapi packages? (Likely requires additional trust config.)
5. **Dual Node/Bun package.json exports:** Should we export CJS + ESM + TypeScript? Or Bun-only? (Affects downstream consumers using Node.)
